# iSub — 隐私模型(Privacy)

最后更新:2026-06-13 ｜ 受众:团队 + 评委 + 接入方
**立场一句话:我们做「不可关联性(unlinkability)」,不做「匿名 / 混币(anonymity)」。** 前者保护用户、且与「非托管 = 监管轻」一脉相承;后者是合规毒药,且对我们的场景多余。

---

## 0. 触发场景

用户想用加密订阅一个**敏感服务**(如成人内容),担心:「认识我钱包的人,会在链上看到我反复给某个已知的 OnlyFans 地址打钱,从而把**我本人**和这个订阅对上。」

这是真实、正当的需求 —— 但正确解法**不是混币**。下面拆清楚。

## 1. 公链上,一次订阅泄漏了什么(威胁模型)

iSub 非托管 + 全链上,代价是默认**可观测**。具体泄漏(对照合约的事件 / 对象):

- `Account<T>.owner` —— 订阅者地址,公开。
- `MandateAuthorized { account_id, subscriber, merchant }` —— 一次性把「某账户授权了某商家」写上链。
- `Charged { account_id, amount, spent_total, seq, by }` + `charge` 内的 `public_transfer(coin, merchant)` —— 每期的金额、时点、收款人,全部公开。

→ 指纹 = **`订阅者地址` ↔ `商家(OnlyFans)地址` ↔ `定额 + 定期`**。危险来自两处:
1. 订阅者地址若是其**主钱包**(挂着 NFT / DeFi 历史 / 域名)→ 直接暴露身份;
2. 「反复打给某个已知地址」本身就是强关联信号,哪怕地址本身匿名。

## 2. 两种「隐私」必须分清

| | 不可关联(unlinkability) | 匿名 / 混币(anonymity) |
|---|---|---|
| 目标 | 旁观者无法把**我本人 / 主钱包**与这笔订阅对上 | 资金流向本身不可追踪 |
| 手段 | 地址隔离、zkLogin、隐形地址 | 混币池、屏蔽转账 |
| 合规 | ✅ 用户保护,监管中性 | ❌ OFAC / AML 重灾区(Tornado Cash 已被制裁) |
| 与我们架构 | ✅ 兼容(可审计 / 对账不破) | ❌ 冲突(归属性是我们多项功能的前提) |

**用户的真实诉求几乎总是第一种**(「别把我对上」),不是第二种(「让钱凭空消失」)。我们据此选型。

## 3. 分层方案

### Tier 0 —— 现在就能交付(集成层,零协议改动)

**专用订阅账户(burner Account)+ zkLogin + 充值卫生。**

- iSub 的 `Account` 是按地址的对象,一个人可开任意多个 → 用一个**全新、无身份**的地址开 Account、`authorize`、订阅,**主钱包永不沾边**。
- **zkLogin**(Sui 原生):用 web2 登录生成新地址,链上不挂持久身份。
  > 注意:zkLogin 是**登录 / 身份**隐私,**不是**转账隐私 —— 它帮你低成本造干净地址,不屏蔽任何转账。别混为一谈。
- **充值卫生(关键残余)**:别从主钱包直接转账给 burner(否则 `主 → burner → OnlyFans` 重新连上)。正确做法:交易所提现 / 法币 on-ramp **直接充值**该地址 —— 中心化那一跳天然断开链上痕迹。
- 效果:挖主钱包 → 一无所获;挖到 burner → 只见「一个干净地址订了 OnlyFans」,**对不上人**。**覆盖绝大多数需求。**

### Tier 1 —— Roadmap(协议 / 集成层,真·密码学,**非混币**)

**商家收款用隐形地址(stealth address)。**

- 商家发布一个「隐形元地址」;**每笔扣款派生一个一次性收款地址**,只有商家能认领并归集。
- 链上再无「反复 → 同一个 OnlyFans 地址」的指纹;旁观者**根本看不出这些是 OnlyFans**。
- 它只是**地址派生**,**不混淆、不池化资金** → **不触 AML**;商家照常收款、照常对账。这是「周期支付隐私」在密码学上的正确原语(类比以太坊 ERC-5564)。
- 完整版还需让 `authorize` 绑定的 `merchant` 也用**每订阅者派生地址**(否则那一次 `MandateAuthorized` 仍暴露 OnlyFans)。
- 代价:密钥派生 + 商家扫描归集;Sui 上**尚无成熟标准,需自建**。**这是值得做的差异化隐私功能。**

### Tier 2 —— 更远(可选)

**保密金额(confidential amounts)**:用承诺 / ZK 隐藏「扣了多少」。隐藏金额、不隐藏关联,较重,且通常不是核心诉求。列待评估。

### 非目标 —— 混币 / 屏蔽池(**明确不做**)

- **合规毒**:把我们「非托管 = 不碰钱 = 监管轻」的命根子炸掉,尤其叠加敏感垂类。
- **毁功能**:可审计扣款、对账、dunning、「凭付款开权限」全部依赖归属性。
- **多余**:商家在「开权限」层本就知道是谁付的;隐私目标是对**公链旁观者**保密,混币用不上。
- **无原生**:Sui 无原生屏蔽转账;得外挂不成熟的第三方协议,徒增攻击面与依赖风险。

## 4. 诚实边界(残余泄漏)

即便做到 Tier 0 + 1,仍存在:

- **充值链**:burner 的资金来源若可追溯,关联可被重建 → 靠 on-ramp / CEX 一跳缓解,**非协议保证**。
- **`MandateAuthorized` 一次性披露**:不做 Tier 1 的派生 merchant 时,`authorize` 那一刻「账户 ↔ OnlyFans」上链一次。
- **时序关联**:固定周期 + 固定金额,即便换地址也可能被聚类分析关联 → 隐形地址 + 金额 / 时点抖动可**缓解**,非消除。
- **商家必然知道**:access 层商家知道是谁(要发内容)。我们保护的是「对公链」,**不是「对商家」**。
- **iSub 不碰钱**:隐私姿态由**用户的地址选择**与**商家的暴露选择**决定;iSub 提供原语,不强制、也无法强制某种隐私级别。

## 5. 合规姿态(写进 pitch)

> iSub 刻意选择**不可关联**而非**匿名**:用户得到实质隐私保护,而我们不变成洗钱基础设施。这与「非托管 = 监管轻」同源 —— 是**合规成熟度**的体现,而非缺陷。能保护用户、又不引来监管火力,这本身是差异化。

## 6. 与「内容访问」隐私的关系(别混淆)

订阅的**内容**访问是另一层:可用 **Seal**(Mysten 的门槛加密 / 链上访问控制)对内容加密、凭订阅状态解密。那是**数据门控**,与**支付隐私正交**;本文只讲支付侧。两者解耦是好事 —— 链上只管「付没付」,内容平台管「给谁看」。

## 7. 集成配方(给接入方 / 给用户)

敏感订阅的推荐姿势:

1. 用 zkLogin 起一个**专用订阅地址**(非主钱包)。
2. 从 CEX 提现 / 法币 on-ramp **直接充值**该地址(切勿从主钱包转)。
3. 用该账户 `authorize` + 订阅;主钱包始终不参与。
4. (Tier 1 上线后)商家启用隐形收款地址 → 连「是 OnlyFans」都看不出。
5. 商家在产品内做 access 门控(可配 Seal),与链上支付解耦。
