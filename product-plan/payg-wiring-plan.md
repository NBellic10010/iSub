# PAYG 打通方案 —— agent 自主按用量付费(端到端)

> 给组内评审 ｜ 目标:把"一堆各自为真的零件"接成"一条真能跑的链路",并在 testnet 上真扣一次。
> 最后更新:2026-06-16

## 1. 一句话
agent 订阅一个按量付费服务后,链上确实建了授权(mandate),但**之后没人接着 billing**——授权 id 只躺在内存里,没有任何运行的服务去计量、去扣款。本方案把这最后一段接通。

## 2. 现状 vs 目标(大白话)

**现状**:agent 签了"授权书"(真的、链上有),然后……没有然后。
- 计量和扣款的代码都写好、单独测过了,但**没接成一个跑着的服务**;
- 扣款引擎(biller)的编排逻辑只对"假链(mock)"测过,**没在真链上跑过**;
- agent 半边和服务半边是两座孤岛,只靠一个 mandate id 概念上连着。

**目标**:
```
agent 订阅 → 链上建 mandate → 把 mandate id 告诉服务(登记)
服务:每次被调用就记一笔用量 → 后台循环到点,把累计用量从 agent 账户扣掉(真链)
全程 agent 不再签名;不重扣、不超上限。
```

## 3. 方案总览(谁干什么)

两个角色:
- **agent 侧(需求方)**:决定订阅、签一次授权。**已有。**
- **服务侧(供给方/商家)**:跑一个"计费服务" = 三样打包:**计量入口 + 扣款引擎(biller) + 数据库**。**这是要新建的东西。**

一个关键安全点:**登记不是盲信**。服务收到一个 mandate id,会**上链核对**——这张授权是真的吗?是付给我的吗?扣款人是我吗?对得上才接。

## 4. 需要组里拍板的决策(评审重点)

| # | 决策 | 选项 | 建议 |
|---|---|---|---|
| D1 | agent 怎么告诉服务有新订阅 | A 主动登记(push) / B 服务扫链上事件(pull) | **push** —— gRPC 没有事件查询;push 简单可靠,真伪由"上链核对"兜底 |
| D2 | 服务怎么部署 | 商家自嵌 / 我们托管 | 先做一个**服务 runner(可嵌可托管)**,两种都能用,先打通 |
| D3 | HTTP 用什么 | 零依赖 `node:http` / 装 Hono | **node:http** —— 零安装、能 headless 测、和现有 `node:sqlite` 一个路子 |
| D4 | 扣款 gas 谁付 | 商家的 keeper key / sponsored | 先商家自付(现状),后续上 sponsored 免 gas |
| D5 | 计量怎么喂进来 | 服务内嵌直接调 / 外部 `POST /usage` | **两个都支持**:嵌入直接调,托管走 HTTP |

## 5. 范围

**做**:
- 服务 runner:计量入口 + biller + DB + 真链 client + 后台 flush 循环
- 两个端点:`POST /subscriptions`(登记,带上链核对)、`POST /usage`(用量)
- api-key 鉴权(已有 merchants 表)
- **一条 testnet 端到端测试**:agent 订阅 → 登记 → 用量 → 真扣

**不做(本轮)**:商家后台 dashboard、webhook 投递串联(已建,后接)、x402 适配、软风控(异常熔断/审批,单独立项)、多租户运维。

## 6. 风险与对策

- **测试网不稳** → demo 备 localnet 一键回退(现成)。
- **biller 第一次上真链可能冒出 mock 没暴露的问题**(时序 / gas / 读链延迟)→ 这正是这条 integration 测试的目的:**现在就把真链问题逼出来**,而不是上线才发现。
- **盲信登记** → 上链核对(真伪 / 收款人 / 扣款人)。
- **重扣** → biller 已两轮对抗审查 + lost-ack/崩溃恢复;这次在真链上再验一次。

## 7. 验收标准(满足才算"打通")

一条命令在 testnet 上跑出:
1. agent(签名 A)订阅一个 PAYG 服务 → 登记成功(服务上链核对通过)
2. 服务记 3 笔用量 → 后台 flush → **真的从 agent 账户扣款给商家**(金额=用量、在上限内)
3. 全程 **agent 的 key 没签任何扣款**(由服务的 key 签)
4. 不重扣、不超 `total_budget`/`rate_cap`、过期/撤销即停
5. 给出 explorer 链接,逐笔可查

## 9. 待讨论的开放问题

1. **登记时机**:agent 自己登记,还是 agent 的宿主平台代登记?
2. **计量信任**:服务自报用量(封顶信任,F-05)——本轮接受,attestation 单独立项。
3. **一个 mandate 一个商家**:agent 付多个服务 = 多张 mandate,确认产品上可接受。
4. **托管 vs 自嵌优先级**:先打通哪种形态?(建议先自嵌 runner,因为它同时是托管的内核。)
